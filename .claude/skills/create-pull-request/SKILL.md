# Pull Request 作成スキル

未コミットの変更を論理的なコミット単位に分割し、`.github/PULL_REQUEST_TEMPLATE.md` に従ったPRを作成するワークフロー。

## ワークフロー

```
Step 1: 変更の把握 + 規模チェック
Step 2: ブランチの確認・作成
Step 3: 差分の分析 + 安全チェック → コミット
Step 4: PRドラフト生成
Step 5: ユーザー確認 → PR作成
```

> 分割する場合は、確定したグループごとにStep 2〜5を繰り返す。

---

## Step 1: 変更の把握 + 規模チェック

`git status` と `git diff HEAD --stat` を実行し、変更ファイル数と変更行数を確認する。

**変更がない場合**: ユーザーに伝えて終了。

**以下のいずれかに該当する場合、PR分割をユーザーに提案する:**
- 変更行数が **400行超**
- 変更ファイル数が **15ファイル超**

分割提案は `references/commit-grouping.md` の「分割提案フォーマット」に従う。

---

## Step 2: ブランチの確認

現在のブランチをユーザーに確認する：

```
現在のブランチは `<branch名>` です。新しいブランチを作成しますか？
それともこのブランチのままPRを作成しますか？
```

- 新しいブランチを作る場合：`feat-<機能名>` / `fix-<修正内容>` などを提案し作成する
- `main` ブランチのままの場合：直接作業を避けるよう警告した上でユーザーの意向に従う

---

## Step 3: 差分の分析と安全チェック → コミット

### 3-1: 新規ファイルの安全チェック

`git status` で新規追加ファイル（`??` または `A`）を特定し、**必ずファイルの内容を読み込む**。

CRITICAL: 以下に該当する場合は、コミット前にユーザーへ確認する:
- 機密情報（`.env`、APIキー、パスワード、トークン等）が含まれている
- 自動生成ファイル（`*.pb.go`、`go.sum`、`package-lock.json`等）が意図せず含まれている
- `vendor/`、`node_modules/`等の依存関係ディレクトリが含まれている
- ファイルの目的・内容が変更の意図と一致していない

確認フォーマットは `references/pr-template-guide.md` の「違和感のある差分の確認フォーマット」を参照。

### 3-2: 差分の分類

`git diff HEAD` を読み込み、変更を **意味のある単位** に分類する。

分類の考え方:
- **目的が異なるもの** → 別コミット（例: バグ修正 vs 新機能）
- **影響範囲が独立しているもの** → 別コミット（例: UI変更 vs ロジック変更）
- **リファクタリング** → 機能変更と分離

コミットメッセージ形式:

```
<type>: <概要>
```

type: `feat` / `fix` / `refactor` / `docs` / `test` / `chore`

**例:**

| 変更内容 | コミットメッセージ |
|---|---|
| ログインAPIにバリデーション追加 | `feat: ログインAPIにリクエストバリデーションを追加` |
| READMEのセットアップ手順を更新 | `docs: セットアップ手順を最新の環境に合わせて更新` |
| 未使用のimportを整理 | `refactor: 未使用のimportを削除` |
| CIのGoバージョンを更新 | `chore: CIのGoバージョンを1.22に更新` |

### 3-3: コミット実行

分類が決まったら **確認なしで自律的に** コミットを実行する。
完了後、作成したコミット一覧をユーザーに表示する。

---

## Step 4: PRドラフトの生成

- `.github/PULL_REQUEST_TEMPLATE.md` が存在する場合はそれに厳密に従う
- ユーザーがカスタムテンプレートを渡してきた場合は、そのテンプレートに従う
- テンプレートがない場合は `references/pr-template-guide.md` のデフォルトテンプレートを使用する

**タイトル**: コミット内容から自動生成する。複数コミットがある場合は全体を俯瞰した簡潔なタイトルにする。

---

## Step 5: ユーザー確認 → PR作成

PRドラフトをユーザーに提示する：

```
【PR ドラフト】

タイトル: <タイトル>

本文:
<本文>

このPRを作成しますか？（修正したい部分があれば教えてください）
```

- ユーザーが承認したら `gh pr create` で作成する
- 修正要望があれば内容を更新して再度確認する

---

## Common Issues

**gh コマンドが見つからない**
- Cause: GitHub CLI がインストールされていない、または未認証
- Solution: `brew install gh` → `gh auth login` で認証する

**コミットが空になる**
- Cause: ファイルがステージされていない
- Solution: `git add <ファイル>` でステージ後に再実行

**PR作成時にブランチが見つからないエラー**
- Cause: ローカルブランチがリモートにプッシュされていない
- Solution: `git push -u origin <branch名>` を実行してから `gh pr create`

## 使用例

**例1: 通常のPR作成**

```
ユーザー: PRを作成して
→ git status で変更を確認（例: 5ファイル、120行）
→ ブランチ確認 → feat-login-validation を作成
→ コミット分類・実行（2コミット）
→ PRドラフト提示 → 承認 → gh pr create でPR作成完了
```

**例2: 大規模変更の分割**

```
ユーザー: 差分をコミットしてPR作ってください
→ 500行超を検出 → 分割を提案
→ ユーザーが承認 → PR1（バックエンド）→ PR2（フロントエンド）の順で作成
```

---

## 注意事項

CRITICAL:
- `main` ブランチへの直接コミット・プッシュは禁止
- `.env` や認証情報を含むファイルはコミットしない
- コミット前に動作確認済みであることを前提とする（不明な場合はユーザーに確認を促す）
- 品質を優先する。検証ステップをスキップしない
