# Pull Request 作成スキル

未コミットの変更を論理的なコミット単位に分割し、`.github/PULL_REQUEST_TEMPLATE.md`に従ったPRを作成する。

## 使用方法

```bash
/create-pull-request
```

## ワークフロー概要

1. 変更の把握 + 規模チェック（大きすぎる場合は分割を提案）
2. ブランチの確認・作成
3. 差分の分析（違和感のある差分はユーザー確認）→ 自律的コミット
4. PRドラフト生成（テンプレートに従う）
5. ユーザー確認 → PR作成

> 分割する場合は各グループごとにStep 2〜5を繰り返す。

---

## Step 1: 変更の把握 + 規模チェック

まず現在の変更状況を確認する。

```bash
git status
git diff HEAD --stat
```

変更がない場合はユーザーに伝えて終了。

### 規模チェック

以下のコマンドで変更行数とファイル数を計測する。

```bash
# 変更ファイル数
git diff HEAD --name-only | wc -l

# 変更行数（追加 + 削除）
git diff HEAD --numstat | awk '{add+=$1; del+=$2} END {print add+del}'
```

**以下のいずれかに該当する場合、PR分割をユーザーに提案する：**

- 変更行数が **400行超**
- 変更ファイル数が **15ファイル超**

#### 分割提案フォーマット

```
変更量が大きいため、PRを分割することを検討してください。

現在の規模:
- 変更ファイル数: <N> ファイル
- 変更行数: <N> 行

以下の分割案を提案します：

【PR 1】<論理的な名前>
  目的: <このPRが達成すること>
  対象ファイル:
  - <ファイルA>
  - <ファイルB>

【PR 2】<論理的な名前>
  目的: <このPRが達成すること>
  対象ファイル:
  - <ファイルC>
  - <ファイルD>

（必要であれば3つ以上に分割）

このまま1つのPRとして作成しますか？それとも分割しますか？
分割する場合、上記の案で進めますか？変更したい点があれば教えてください。
```

#### 分割案の作り方

分割は以下の観点で **論理的に意味のあるまとまり** を作る：

- **レイヤー単位**：インフラ・DB・バックエンド・フロントエンドを分ける
- **機能単位**：機能Aと機能Bが独立していれば別PR
- **種別単位**：リファクタリングは機能追加と分離する
- **依存関係順**：後のPRが前のPRに依存する場合、その順番を明記する

ユーザーが分割を選んだ場合、確定したグループごとに **Step 2〜Step 5を繰り返す**。
ユーザーが分割しないと判断した場合はそのまま次のステップへ進む。

---

## Step 2: ブランチの確認

**必ずユーザーに確認する（自動で判断しない）：**

> 現在のブランチは `<branch名>` です。新しいブランチを作成しますか？それともこのブランチのままPRを作成しますか？

- 新しいブランチを作る場合：`feat-<機能名>` や `fix-<修正内容>` など適切な名前を提案し、作成する
- そのままの場合：`main` ブランチでの直接作業は避けるよう警告した上でユーザーの意向に従う

---

## Step 3: 差分の分析と自律的コミット

`git diff HEAD` の内容を読み込み、変更を **意味のある単位** に分類する。

> **注意**: `git diff HEAD` は既存ファイルへの変更のみを表示し、新規追加ファイル（untracked または staged new file）の内容は表示されない。新規ファイルが含まれる場合は、後述の「新規追加ファイルの内容確認」を必ず実施すること。

### 分類の考え方

変更は以下の軸で分類する：

- **目的が異なるもの**を別コミットにする（例：バグ修正 vs 新機能追加）
- **影響範囲が独立しているもの**を別コミットにする（例：UI変更 vs ロジック変更）
- **リファクタリングは機能変更と分離**する

### コミットメッセージ形式

```
<type>: <概要>
```

type: `feat` / `fix` / `refactor` / `docs` / `test` / `chore`

### 新規追加ファイルの内容確認

`git status` で新規追加ファイル（`??` または `A`）を特定し、**必ずファイルの内容を読み込んで確認する**。

```bash
# 新規追加ファイルの一覧を取得
git status --short | grep -E '^(\?\?|A )'
```

確認する観点：

- `.env`、APIキー、パスワード、トークン等の機密情報が含まれていないか
- 自動生成ファイル（`*.pb.go`、`go.sum`、`package-lock.json` 等）が意図せず含まれていないか
- `vendor/`、`node_modules/` 等の依存関係ディレクトリが含まれていないか
- ファイルの目的・内容が変更の意図と一致しているか

問題がある場合は「違和感のある差分の検出」と同じフォーマットでユーザーに確認する。

---

### 違和感のある差分の検出

分類前に、以下のいずれかに該当する変更がないか確認する。該当があれば **コミット前にユーザーへ確認**する。

**確認が必要なケース：**

- 変更の目的と無関係に見えるファイルが含まれる
- 大量の削除・ファイル丸ごとの削除
- バイナリファイルや自動生成ファイル（`package-lock.json`、`go.sum`、`*.pb.go` 等）への予期しない変更
- `.env`、認証情報、シークレットを含む可能性のあるファイル
- `vendor/`、`node_modules/` など依存関係ディレクトリの変更

**確認フォーマット：**

```
以下の変更が含まれていますが、意図したものですか？

- <ファイル名>: <変更内容の概要>
（他にあれば列挙）

コミットに含めますか？含めない場合は除外します。
```

ユーザーが除外を選んだファイルは `git restore` 等で除外してからコミットに進む。
違和感のある差分がない場合はそのまま次のステップへ進む。

### コミット実行

分類が決まったら **確認なしで自律的に** コミットを実行する。

```bash
git add <対象ファイル>
git commit -m "<コミットメッセージ>"
```

コミット完了後、作成したコミット一覧をユーザーに表示する。

---

## Step 4: PRドラフトの生成

### テンプレートが提供されている場合

ユーザーが「このフォーマットで」とテンプレートを渡してきた場合は、そのテンプレートに厳密に従ってPRの本文を生成する。

### テンプレートがない場合

以下のデフォルトテンプレートを使用する：

```
## 概要
<!-- この PR で何をしたかを簡潔に -->

## 変更内容
<!-- 箇条書きで変更点を列挙 -->

## 動作確認
<!-- 確認した内容を記載 -->
```

### タイトル

コミット内容から自動生成する。複数コミットがある場合は全体を俯瞰した簡潔なタイトルにする。

---

## Step 5: ユーザー確認 → PR作成

生成したPRドラフトを表示する：

```
【PR ドラフト】

タイトル: <タイトル>

本文:
<本文>

このPRを作成しますか？（修正したい部分があれば教えてください）
```

ユーザーが承認したら `gh pr create` で作成する：

```bash
git push -u origin <branch名>
gh pr create --title "<タイトル>" --body "$(cat <<'EOF'
<本文>
EOF
)"
```

修正要望があれば内容を更新して再度確認する。

---

## 注意事項

- `main` ブランチへの直接コミット・プッシュは避ける
- コミット前に動作確認済みであることが前提（ユーザーに確認を促す）
- `.env` や認証情報を含むファイルはコミットしない
