# Pull Request 作成スキル

未コミットの変更を論理的なコミット単位に分割し、`.github/PULL_REQUEST_TEMPLATE.md`に従ったPRを作成する。

## 使用方法

```bash
/create-pull-request
```

## ワークフロー概要

以下のチェックリストに沿って進行する：

```
- [ ] Step 1: 変更の把握 + 規模チェック
- [ ] Step 2: ブランチの確認・作成
- [ ] Step 3: 差分の分析（違和感チェック含む） → 自律的コミット
- [ ] Step 4: PRドラフト生成
- [ ] Step 5: ユーザー確認 → PR作成
```

> 分割する場合は各グループごとにStep 2〜5を繰り返す。

---

## Step 1: 変更の把握 + 規模チェック

変更ファイル数と変更行数（追加+削除）を確認する。変更がない場合はユーザーに伝えて終了。

### 規模チェック

**以下のいずれかに該当する場合、PR分割をユーザーに提案する：**

<!-- レビュー負荷の研究（Google "Modern Code Review" 等）では、200〜400行を超えるとレビュー品質が低下する。安全マージンを取り400行を閾値とする -->
- 変更行数が **400行超**
<!-- 15ファイル超は複数の関心事が混在している可能性が高い。経験的にレビュアーがコンテキストを保持できる上限 -->
- 変更ファイル数が **15ファイル超**

分割を提案する場合は [commit-grouping.md](./commit-grouping.md) の「分割提案フォーマット」に従う。

---

## Step 2: ブランチの確認

**必ずユーザーに確認する（自動で判断しない）：**

> 現在のブランチは `<branch名>` です。新しいブランチを作成しますか？それともこのブランチのままPRを作成しますか？

- 新しいブランチを作る場合：`feat-<機能名>` や `fix-<修正内容>` など適切な名前を提案し、作成する
- そのままの場合：`main` ブランチでの直接作業は避けるよう警告した上でユーザーの意向に従う

---

## Step 3: 差分の分析と自律的コミット

`git diff HEAD` の内容を読み込み、変更を **意味のある単位** に分類する。

> **注意**: `git diff HEAD` は新規追加ファイル（untracked）の内容を表示しない。新規ファイルが含まれる場合は「新規追加ファイルの内容確認」を必ず実施すること。

### 分類の考え方

- **目的が異なるもの**を別コミットにする（例：バグ修正 vs 新機能追加）
- **影響範囲が独立しているもの**を別コミットにする（例：UI変更 vs ロジック変更）
- **リファクタリングは機能変更と分離**する

### コミットメッセージ形式

```
<type>: <概要>
```

type: `feat` / `fix` / `refactor` / `docs` / `test` / `chore`

**例：**

| 変更内容 | コミットメッセージ |
|---|---|
| ログイン API にバリデーション追加 | `feat: ログインAPIにリクエストバリデーションを追加` |
| README のセットアップ手順を更新 | `docs: セットアップ手順を最新の環境に合わせて更新` |
| 未使用の import を整理 | `refactor: 未使用のimportを削除` |
| CI の Go バージョンを更新 | `chore: CIのGoバージョンを1.22に更新` |

### 新規追加ファイルの内容確認

`git status` で新規追加ファイル（`??` または `A`）を特定し、**必ずファイルの内容を読み込んで確認する**。

確認する観点：

- 機密情報（`.env`、APIキー、パスワード、トークン等）が含まれていないか
- 自動生成ファイル（`*.pb.go`、`go.sum`、`package-lock.json` 等）が意図せず含まれていないか
- `vendor/`、`node_modules/` 等の依存関係ディレクトリが含まれていないか
- ファイルの目的・内容が変更の意図と一致しているか

問題がある場合は [pr-template-guide.md](./pr-template-guide.md) の「違和感のある差分の確認フォーマット」でユーザーに確認する。

---

### 違和感のある差分の検出

分類前に、以下のいずれかに該当する変更がないか確認する。該当があれば **コミット前にユーザーへ確認**する。

**確認が必要なケース：**

- 変更の目的と無関係に見えるファイルが含まれる
- 大量の削除・ファイル丸ごとの削除
- バイナリファイルや自動生成ファイルへの予期しない変更
- `.env`、認証情報、シークレットを含む可能性のあるファイル
- `vendor/`、`node_modules/` など依存関係ディレクトリの変更

確認フォーマットは [pr-template-guide.md](./pr-template-guide.md) の「違和感のある差分の確認フォーマット」を参照。

ユーザーが除外を選んだファイルは除外してからコミットに進む。違和感のある差分がない場合はそのまま次のステップへ進む。

### コミット実行

分類が決まったら **確認なしで自律的に** コミットを実行する。コミット完了後、作成したコミット一覧をユーザーに表示する。

---

## Step 4: PRドラフトの生成

### テンプレートが提供されている場合

ユーザーが「このフォーマットで」とテンプレートを渡してきた場合は、そのテンプレートに厳密に従ってPRの本文を生成する。

### テンプレートがない場合

[pr-template-guide.md](./pr-template-guide.md) のデフォルトテンプレートを使用する。

### タイトル

コミット内容から自動生成する。複数コミットがある場合は全体を俯瞰した簡潔なタイトルにする。

---

## Step 5: ユーザー確認 → PR作成

生成したPRドラフトを表示する：

```
【PR ドラフト】

タイトル: <タイトル>

本文:
<本文>

このPRを作成しますか？（修正したい部分があれば教えてください）
```

ユーザーが承認したら `gh pr create` で作成する。修正要望があれば内容を更新して再度確認する。

---

## 注意事項

- `main` ブランチへの直接コミット・プッシュは避ける
- コミット前に動作確認済みであることが前提（ユーザーに確認を促す）
- `.env` や認証情報を含むファイルはコミットしない
